STAQTAPP 1.x.x — ORIGINAL ENGINE OVERHAUL ROADMAP
=================================================

Document status
---------------
Purpose: Define the controlled rescue, reconstruction, and completion of the
original Staqtapp Python implementation while retaining the Staqtapp 1.x.x
product line and the original public API focus.

Version policy: The overhaul remains within Staqtapp 1.x.x. A 2.0 designation
is not justified by internal reconstruction alone. Version 2.0 is reserved for
a deliberate, documented break in the public API contract or abandonment of
compatibility with existing SQTPP files.

Milestone notation: This roadmap uses symbolic releases such as 1.A.x, 1.B.x,
and 1.C.x because the original module does not expose a reliable
machine-readable version declaration. Exact minor and patch numbers must be
assigned after the baseline repository and release-history audit.

Primary objective
-----------------
Preserve the recognizable Staqtapp API and SQTPP storage concepts while
replacing the monolithic, fragile implementation with a typed, transactional,
profile-driven engine.

The original module will remain available as:

1. A historical behavior reference.
2. A legacy SQTPP parser and migration source.
3. A source of feature names and intended semantics.
4. A compatibility oracle for verified behavior.
5. A catalogue of incomplete and experimental feature designs.

The original monolith will not remain the production engine.

NON-NEGOTIABLE ENGINEERING RULES
================================

1. Remain in the Staqtapp 1.x.x product line.

2. Preserve the original API focus:
   - Public function names.
   - Positional argument order.
   - Default argument behavior where defined.
   - Return shapes for verified, valid calls.
   - Existing supported SQTPP file behavior.
   - Familiar module-level calling patterns.

3. Do not preserve bugs for compatibility:
   - Silent corruption becomes an explicit error.
   - Ambiguous success becomes a typed result or typed exception.
   - Unfinished operations raise NotImplementedError until completed.
   - Unsafe execution is isolated or redesigned.

4. Preserve source SQTPP files during migration:
   - Never destructively upgrade the only copy.
   - Create a migrated or canonicalized destination.
   - Produce a machine-readable migration report.
   - Verify record counts, hierarchy, values, locks, and history.

5. Optimize from evidence:
   - Profile before changing hot paths.
   - Benchmark every claimed performance improvement.
   - Track CPU, allocation, I/O, lock, and process costs.
   - Reject ornamental use of mmap, memoryview, deque, or multiprocessing.

6. Reads are buffer-oriented:
   - Bytes remain bytes until interpretation is required.
   - Offsets remain offsets until materialization is required.
   - Avoid whole-file string decoding for ordinary lookup.
   - Prefer read-only mmap snapshots and memoryview slices.

7. Writes are transactional:
   - No in-place variable-length mutation of live mapped files.
   - Write to a temporary file.
   - Flush, fsync, and atomically replace.
   - Preserve recoverable backups according to policy.
   - Detect conflicting revisions.

8. Diagnostics do not control storage:
   - Storage emits bounded counters and immutable events.
   - Diagnostic queues may drop events under pressure.
   - Diagnostic failure must not block or corrupt storage operations.

9. Multiprocessing is explicit:
   - Top-level pickleable API wrappers remain available.
   - Readers may operate concurrently on read-only mappings.
   - Writes to one VFS are serialized through a transaction coordinator.
   - Pool orchestration is not misrepresented as parallel write throughput.

10. Advanced features return only after contracts and tests exist:
    - Pojishon.
    - Registry and schemas.
    - Lambda facilities.
    - NTR/tree facilities.
    - Dark variables.
    - Genic and vital variables.
    - Revar semantics.
    - Any generated-code execution path.

Compatibility facade
--------------------
The public facade remains recognizable:

    makevfs(...)
    setpath(...)
    addvar(...)
    appvar(...)
    loadvar(...)
    changevar(...)
    renamevar_stx(...)
    removevar(...)
    listvars(...)
    listfiles(...)
    ...

The facade performs:

1. Argument normalization.
2. Compatibility handling.
3. Validation.
4. Dispatch into the typed engine.
5. Return-shape adaptation.
6. Deprecation warnings where necessary.
7. Typed exception translation.

The facade does not perform raw file parsing or raw string splicing.

Typed VFS model
---------------
The internal model must represent:

- VFS identity.
- Format version.
- Directory.
- Folder.
- Subfolder.
- Embedded file.
- Variable.
- Variable type.
- Variable payload span.
- Lock.
- Stalk/history record.
- Tree record.
- Registry record.
- Feature-extension record.
- Unknown legacy record.

Unknown records must be preserved where possible so canonical rewrites do not
silently destroy data from unsupported legacy features.

Storage format layer
--------------------
The format module defines all structural markers, delimiters, escaping rules,
record boundaries, field encodings, checksums, revision fields, and extension
records.

No feature module may independently hard-code format markers.

Parser
------
The parser has two levels:

Level 1: Structural scan
- Operates on mmap-backed bytes.
- Finds record spans and type identifiers.
- Builds compact indexes.
- Avoids eager string decoding.
- Preserves unknown spans.

Level 2: Lazy materialization
- Decodes names and values only when requested.
- Uses memoryview slices for payload access.
- Produces typed objects only as needed.
- Caches decoded values within bounded limits.

Serializer
----------
The serializer:

- Produces one canonical representation for each supported record.
- Streams large unchanged ranges where practical.
- Preserves unsupported records when safe.
- Rejects irreconcilably malformed structures.
- Calculates checksums and revision metadata.
- Never edits a live file with uncontrolled string replacement.

Transaction layer
-----------------
Every mutation follows:

1. Acquire process-safe VFS write lock.
2. Confirm file identity and expected revision.
3. Open a read-only snapshot.
4. Parse or load a validated index.
5. Construct a mutation plan.
6. Write a temporary destination.
7. Flush user-space buffers.
8. fsync the destination.
9. Atomically replace the active VFS.
10. fsync the containing directory where supported.
11. Publish the new revision.
12. Invalidate stale mappings and indexes.
13. Release the lock.
14. Emit a diagnostic event outside the critical section where possible.

A crash before atomic replacement leaves the original intact. Recovery tooling
handles orphan temporary files and validated backups.


PERFORMANCE ARCHITECTURE
========================

mmap
----
Use read-only mmap for:

- Initial structural scans.
- Variable lookup.
- Directory listing.
- Embedded-file reads.
- Integrity checks.
- Migration scans.
- Building offset indexes.
- Sharing operating-system page cache across reader processes.

Do not use writable mmap as the general mutation mechanism for variable-length
records.

A mapped snapshot is invalidated when any of these change:

- Device identifier.
- Inode or file identity.
- File size.
- Modification time in nanoseconds.
- Internal revision.
- Format version.

memoryview
----------
Use memoryview for:

- Zero-copy record slicing.
- Header decoding.
- Checksum input.
- Payload transfer.
- Codec input.
- Embedded-file extraction.
- struct.unpack_from operations.
- Copying unchanged ranges.

Avoid calling bytes() or decode() inside hot loops unless the public operation
requires an owned or textual result.

deque
-----
Use bounded deque structures for:

- Diagnostic events.
- Recent transaction records.
- Parser work queues.
- Breadth-first tree traversal.
- Deferred mutation plans.
- Recent negative lookups.
- Bounded history caches.

A normal deque is process-local. Cross-process transfer uses a bounded
multiprocessing queue, a dedicated coordinator, or a designed shared-memory
ring.

Other required tools
--------------------

dataclass(slots=True)
- Compact typed records.
- Deterministic fields.
- Lower per-instance overhead.

struct.Struct
- Precompiled binary header parsing.
- Fixed-width metadata.
- Zero-copy unpack_from use.

array
- Compact offset and hash tables where Python lists are too expensive.

bytearray
- Mutable record assembly for small and medium output units.

bisect
- Ordered physical indexes and range searches.

dict
- Exact variable and path lookup unless profiling proves another structure is
  superior.

hashlib
- Buffer-based content and structural checksums.

os.copy_file_range / os.sendfile
- Optional fast copying of unchanged large regions when supported and proven
  beneficial.

sys.intern
- Only for highly repeated structural strings after allocation profiling.

functools.lru_cache
- Only for bounded, pure decoding or schema operations.

Shared memory
- Consider only after measuring process serialization and index rebuild cost.
- Do not introduce before single-process correctness is complete.

Performance doctrine
--------------------
The new engine must report both speed and write amplification.

A fast lookup paired with catastrophic whole-file rewrite behavior is not a
complete performance success. Measurements must include:

- Open latency.
- Cold-cache and warm-cache lookup.
- Index construction time.
- Decode latency.
- Update latency.
- Bytes copied per update.
- Bytes written per update.
- Peak memory.
- Python allocation count.
- Lock hold duration.
- fsync duration.
- Reader scaling.
- Writer contention.
- Process startup and serialization cost.


PUBLIC API RECOVERY MAP
=======================

Tier 1 — Core compatibility surface
-----------------------------------
These functions receive the earliest production-grade implementation:

makevfs
- Create a new VFS with validated initial hierarchy.
- Never overwrite an existing VFS without explicit policy.
- Return compatibility-adapted result.

setpath
- Select an existing VFS hierarchy.
- Validate exact path components.
- Replace global implicit state internally with a context-aware path manager
  while preserving the original call.

addvar
- Add one variable.
- Reject duplicates and malformed names.
- Preserve verified legacy value semantics.

appvar
- Add a validated batch.
- Reject duplicate names in the request.
- Reject collisions with existing variables.
- Preserve intentional null-fill behavior only if confirmed as contract.

loadvar
- Preserve existing valid calling convention and return shape.
- Internally support indexed lazy reads.
- Optional future keyword features may be added without breaking positional
  calls.

changevar
- Transactionally replace variable data.
- Preserve locks and history according to documented semantics.

renamevar_stx
- Rename a normal or stalk-related variable according to explicit mode rules.
- Update indexes and references transactionally.

removevar
- Remove a variable without corrupting adjacent records.
- Respect locks.
- Define history retention or removal behavior.

findvar
- Exact indexed lookup.

findvar_stx
- Search stalk/history structures with specified ordering and result shape.

listvars
- Return variables in documented deterministic order.

vardata_stx
- Literal or regular-expression search.
- Apply bounds and safe regex policy for untrusted patterns.

joinvars
- Specify input ordering, type coercion, collision handling, and source
  retention.

corevar
- Preserve verified bit encoding and decoding.
- Move codec implementation into an isolated module with property tests.

MULTIPROCESSING AND CALLABLE BOTTOM-MODULE API
==============================================

Required public dispatch
------------------------
The module-level facade remains callable by multiprocessing spawn workers.

Required functions:

invoke_api(name, args=(), kwargs=None)
- Dispatch only allow-listed public API names.
- Reject private or arbitrary attribute access.
- Normalize results into pickleable forms.
- Preserve typed error metadata where pool boundaries require it.

map_api_calls(calls, processes=None, start_method="spawn", chunksize=1)
- Accept tuple and mapping call specifications.
- Validate every call before pool creation.
- Use top-level workers only.
- Support ordered result return.
- Document that writes to one VFS remain serialized.

Optional future functions:

imap_api_calls(...)
- Streaming ordered results.

imap_unordered_api_calls(...)
- Streaming unordered results for independent read workloads.

submit_api_call(...)
- Executor-oriented integration.

Bottom-module guard
-------------------
The importable module retains:

    if __name__ == "__main__":
        multiprocessing.freeze_support()

No application code, demonstration mutation, or test execution occurs under
the production module guard.

Process model
-------------
Preferred model:

Many reader processes
- Open read-only mmap snapshots.
- Reuse operating-system page cache.
- Maintain per-process lightweight indexes or load verified sidecar indexes.
- Remap after revision change.

One writer coordinator per VFS
- Serializes mutation plans.
- Holds the shortest practical exclusive lock.
- Applies transaction writes.
- Publishes revision changes.
- Never depends on the diagnostic consumer.

Independent VFS files
- May be written concurrently because their locks and transaction identities
  are separate.

Global path compatibility
-------------------------
The original API uses selected-path state. Internally, the new engine should
support explicit VFS contexts:

    with open_vfs(path) as vfs:
        ...

The compatibility facade may resolve the selected legacy path and pass an
explicit context internally.

This prevents the new core from inheriting unnecessary global state while
preserving old calls.

INDEX DESIGN
============

Initial in-memory index
-----------------------
Each variable or record entry should contain only the data needed for lookup:

- Stable name hash.
- Name offset.
- Name length.
- Record start.
- Record end.
- Payload start.
- Payload length.
- Record type.
- Flags.
- Revision or generation.
- Optional checksum.

Exact lookup
------------
Use a dictionary from normalized name to compact entry identifier unless
profiling demonstrates a better option.

Physical order
--------------
Use a compact ordered offset table for:

- Canonical serialization.
- Range copying.
- Iteration.
- Corruption detection.
- Binary searches over physical spans.

Sidecar index
-------------
A sidecar index is allowed within Staqtapp 1.x.x if:

- It is disposable.
- It contains no unique data.
- It is verified against VFS identity and revision.
- Corruption causes rebuild, not VFS failure.
- Existing SQTPP files remain readable without it.
- Its format is versioned.
- It is written atomically.

Possible suffix: .sqti, subject to naming audit.

Embedded index
--------------
An embedded index may be considered later, but only with a compatible format
extension and migration plan.

SECURITY AND SAFETY
===================

Mandatory changes
-----------------

- No unrestricted pickle loading.
- No arbitrary eval.
- No arbitrary exec.
- No automatic import of generated VFS Python.
- No path traversal.
- No implicit overwrite.
- No regex denial-of-service exposure without controls.
- No silent malformed-record recovery that changes data.
- No unbounded diagnostic queue.
- No unbounded cache.
- No unverified sidecar index use.
- No mutation without revision conflict detection.

Trusted legacy mode
-------------------
A separately named, explicitly enabled migration-only mode may inspect legacy
content that the safe engine refuses. It must:

- Never execute stored content.
- Run with narrow privileges where practical.
- Produce a report.
- Require explicit opt-in.
- Write only to a new destination.

RELEASE AND COMPATIBILITY POLICY
================================

Patch release
-------------
Use 1.x.patch for:

- Defect repairs.
- Security hardening.
- Performance improvements with unchanged supported behavior.
- Documentation.
- Additional diagnostics.
- Compatibility corrections.

Minor release
-------------
Use 1.minor.0 for:

- New stable engine subsystems.
- Complete hierarchy CRUD.
- Migration tooling.
- Reintroduced advanced features.
- New optional API capabilities that preserve existing calls.

Major release
-------------
Do not use 2.0 unless Staqtapp intentionally:

- Breaks the public API contract.
- Drops supported SQTPP compatibility.
- Changes the product model beyond compatibility adaptation.

Internal architecture generation
--------------------------------
Documentation may say:

    Product: Staqtapp 1.x.x
    Engine: Staqtapp Core, second-generation architecture
    Storage: SQTPP legacy-compatible format plus versioned extensions

This does not require a 2.0 product version.

DEFINITION OF DONE
==================

The Staqtapp 1.x.x overhaul is complete when:

1. The original public API focus is preserved through a compatibility facade.

2. The production core no longer depends on the monolithic original
   implementation.

3. Supported SQTPP files can be read, verified, migrated, and rewritten without
   logical data loss.

4. Core variable CRUD is fully transactional and indexed.

5. Directory, folder, subfolder, and embedded-file CRUD is complete.

6. Locks and stalk history have explicit contracts and concurrency tests.

7. Read paths use mmap and memoryview where measurements justify them.

8. Internal record structures are compact and typed.

9. Multiprocessing wrappers are top-level, allow-listed, and spawn-safe.

10. Concurrent readers and serialized writers behave correctly.

11. Every claimed optimization has repeatable benchmark evidence.

12. Diagnostics are bounded and cannot control storage correctness.

13. Arbitrary generated-code execution is absent from default production paths.

14. Advanced features are individually stable, experimental, or formally
    deprecated; none silently pretend to work.

15. Migration never destroys the only source copy.

16. Corruption and interruption tests pass.

17. Cross-platform lock and process behavior is validated on supported systems.

18. The release remains within the Staqtapp 1.x.x product line.

FINAL DIRECTION
===============

Staqtapp is not being discarded and it is not being renamed into an unrelated
2.0 product.

The overhaul preserves:

- The Staqtapp name.
- The 1.x.x version line.
- The original API focus.
- The SQTPP storage identity.
- Verified user-facing behavior.
- The intended advanced feature concepts.

The overhaul replaces:

- Monolithic raw-text mutation.
- Unbounded scratch-state design.
- Silent incomplete behavior.
- Generated-code execution as storage logic.
- Unsafe deserialization.
- Whole-file eager decoding on routine reads.
- Unmeasured performance assumptions.
- Ambiguous concurrency.

The target is a hard-nosed Python storage engine: mmap-backed reads,
memoryview-oriented parsing, compact indexes, transactional writes, bounded
diagnostics, spawn-safe callable wrappers, and a compatibility layer that keeps
the original Staqtapp API recognizable throughout the 1.x.x rescue.
